# Naming Rules

## Обязательно
- Использовать имена, отражающие роль и ответственность.
- Соблюдать единый стиль по проекту (`camelCase`/`snake_case`/`PascalCase`).
- Называть файлы и модули по доменному смыслу, а не по истории изменения.
- Для временных сущностей добавлять явный маркер и срок удаления.

## Запрещено
- Абстрактные имена вроде `data`, `utils2`, `temp-final`.
- Однобуквенные имена вне коротких локальных циклов.
- Смешивание языков и стилей именования в одном контуре.

## Желательно
- Использовать глаголы для действий (`createUser`, `validateInput`).
- Использовать существительные для сущностей (`Order`, `SessionToken`).
- Держать длину имени достаточной для понимания без комментария.
