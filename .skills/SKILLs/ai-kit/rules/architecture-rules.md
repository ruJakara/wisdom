# Architecture Rules

## Обязательно
- Явно разделять слои: интерфейсы, доменная логика, инфраструктура.
- Держать у каждого модуля одну главную ответственность.
- Фиксировать архитектурные решения, влияющие на масштабируемость и поддержку.
- Проектировать контракты до глубокой реализации интеграций.

## Запрещено
- Круговые зависимости между модулями.
- Протаскивание инфраструктурных деталей в бизнес-логику.
- Избыточные абстракции без подтверждённой пользы.

## Желательно
- Предпочитать композицию вместо жёсткой иерархии.
- Держать интерфейсы стабильными, а реализации заменяемыми.
- Планировать точки расширения только для ожидаемых сценариев.
