# Project Recovery

## Title
Project Recovery

## Purpose
Быстро восстановить управляемость полуживого проекта: понять текущее состояние, локализовать поломки и сформировать безопасный план стабилизации.

## Scope
- Диагностика структуры, входных точек и критичных зависимостей.
- Поиск сломанных контуров, мёртвых частей и конфликтующих конфигов.
- Подготовка и реализация минимального recovery-плана.

## Inputs
- Исходный код и конфиги проекта.
- Логи, баг-репорты, наблюдаемые симптомы.
- Доступные команды запуска/сборки/тестов.

## Outputs
- Карта текущего состояния (что живо/что сломано).
- Список первопричин и приоритетов восстановления.
- Набор точечных правок для возврата проекта в рабочее состояние.

## Responsibilities
- Отделять симптомы от причин.
- Восстанавливать сначала критичные пользовательские/бизнес-потоки.
- Фиксировать допущения и неизвестные зоны явно.
- Оставлять проект в воспроизводимом состоянии после каждого шага.

## Constraints
- Не запускать большой рефакторинг во время аварийного восстановления.
- Не менять архитектуру без необходимости стабилизации.
- Не объявлять "исправлено", пока не пройдены проверки ключевых потоков.

## What to avoid
- Случайные правки без карты зависимостей.
- Маскировка ошибок временными заглушками без пометок.
- Восстановление второстепенного, пока не стабилизирован критический путь.

## Typical tasks
- Понять, почему проект перестал запускаться после серии правок.
- Локализовать разрыв между API и клиентом.
- Восстановить базовые сценарии деплоя/сборки/миграций.

## Quality checklist
- Есть список критичных потоков и их текущий статус.
- Первопричины подтверждены, а не угаданы.
- Критичные сценарии снова работают воспроизводимо.
- Долги recovery-этапа отмечены отдельно и не скрыты.
