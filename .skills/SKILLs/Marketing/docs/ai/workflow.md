# Workflow

## Как начинать новую задачу

1. Прочитать `docs/ai/project-map.md`, `docs/ai/architecture.md`, `docs/ai/rules.md`.
2. Уточнить тип задачи: анализ, написание поста или оркестрация процесса.
3. Собрать фактический контекст бренда из запроса/материалов, не додумывая отсутствующее.

## Какие файлы смотреть сначала

- Для управления процессом: `skills/vk-content-director.md`.
- Для аналитики: `skills/vk-audience-analysis.md`.
- Для текстов: `skills/vk-post-writer.md`.
- Для критериев готовности: `docs/ai/done-criteria.md`.

## Как вносить изменения

1. Выбрать минимальный набор файлов под задачу.
2. Изменять прежде всего файл, который владеет этой ответственностью.
3. Сохранять структуру секций в skill-файлах.
4. Если меняется поведение процесса, синхронизировать `architecture.md` и `workflow.md`.

## Как не ломать архитектуру

- Не объединять роли анализа и написания в один этап.
- Не удалять требование маркировать гипотезы.
- Не менять формат выхода skill-файла без явной причины и фиксации в docs.

## Как проверять результат

- Прогнать `task check` (если установлен Task).
- Проверить, что новые/изменённые файлы логически согласованы между собой.
- Убедиться, что в тексте нет противоречий старым правилам без обновления docs.

## Что обновлять в документации

- При смене структуры: `project-map.md`.
- При смене ролей/связей модулей: `architecture.md`.
- При смене практики выполнения задач: `workflow.md` и `rules.md`.
- При смене критериев качества: `done-criteria.md`.

## Как завершать задачу

1. Свериться с `docs/ai/done-criteria.md`.
2. Проверить, что нет незадокументированных новых ограничений.
3. Оставить проект в состоянии, где следующая AI-итерация может стартовать без догадок.
